#+TITLE: gptel-org-agent — Agent Subtrees AI Context
#+DESCRIPTION: Reference document for AI-assisted development of gptel-org-agent.el

* Overview

The gptel-org-agent feature implements an agent subtree system for Org mode.
When =gptel-org-agent-subtrees= is non-nil and =gptel-send= is invoked on a
TODO heading, the feature creates a child heading tagged =:main@agent:=,
opens an indirect buffer narrowed to that subtree, and redirects the LLM
response into the indirect buffer — keeping the base Org buffer clean.

The system supports recursive sub-agent delegation: sub-agents get their own
tagged subtrees (e.g. =:researcher@main@agent:=, =:gatherer@main@agent:=),
each in its own indirect buffer.

** Core Module

- File: =gptel-org-agent.el= (~620 lines)
- Provides: =gptel-org-agent=
- Depends on: =org=, =gptel= (via declare-function)
- Auto-loaded by =gptel-org.el= via =(require 'gptel-org-agent nil t)=
- Feature-flagged: controlled by =gptel-org-agent-subtrees= defcustom (default =nil=)

** Tests

- =test/gptel-org-agent-test.el= — comprehensive tests for all phases
- =test/gptel-org-subtree-test.el= — regression test for branching+agent-subtrees interaction
- Test command: =cd quelpa/build/gptel/test && make test 2>&1 | tail -30=

* Architecture

** Request Flow with Agent Subtrees

#+begin_example
gptel-send on TODO heading
  → gptel-org-agent--transform-redirect (via gptel-prompt-transform-functions)
    ├─ Guard: gptel-org-agent-subtrees non-nil?
    ├─ Guard: point on a TODO heading?
    ├─ Guard: not already in an agent indirect buffer?
    └─ gptel-org-agent--maybe-setup-subtree
       ├─ Find existing :main@agent: child subtree, or create new one
       ├─ gptel-org-agent--create-subtree → marker to new heading
       └─ gptel-org-agent--open-indirect-buffer → indirect buffer
  → FSM :buffer and :position redirected to indirect buffer
  → LLM response streams into indirect buffer (auto-expanding narrow region)
  → Base buffer shows full tree structure, agent subtree foldable
#+end_example

** Context Stripping

During prompt construction, =gptel-org.el= strips =@agent= subtrees from the
prompt copy so the LLM does not see prior agent conversations:

#+begin_src elisp
(when (and (bound-and-true-p gptel-org-agent-subtrees)
           (not (and (boundp 'gptel-org-agent-include-subtrees)
                     (buffer-local-value 'gptel-org-agent-include-subtrees org-buf)))
           (fboundp 'gptel-org-agent--strip-agent-subtrees))
  (gptel-org-agent--strip-agent-subtrees lineage-positions))
#+end_src

In branching context mode, lineage heading positions are protected from
stripping via the =protected-positions= parameter.

** Tool Confirmation Flow (Phase 6)

When a tool call requires user confirmation in subtree mode:

#+begin_src
LLM streams response into indirect buffer
  → TOOL state: gptel--handle-tool-use collects pending-calls
  → Callback dispatches (tool-call . pending-calls)
  → INTERCEPTED by :around advice on gptel--display-tool-calls
    → gptel-org-agent--subtree-tool-confirm-p checks:
      ├─ gptel-org-agent-subtrees enabled?
      ├─ Buffer is org-mode indirect buffer?
      └─ If yes → gptel-org-agent--display-tool-calls
         ├─ Creates child heading: "******** PENDING Requesting permission to run: Bash"
         ├─ Inserts tool call details as body text
         ├─ Stores (pending-id → tool-calls + info) in global hash table
         └─ Stores GPTEL_PENDING_ID property on heading
  → User changes TODO state:
    ├─ PENDING → ALLOWED: org-after-todo-state-change-hook fires
    │  → gptel-org-agent--on-todo-state-change retrieves data
    │  → gptel-org-agent--accept-tool-calls runs tools
    │  → FSM continues with tool results
    └─ PENDING → DENIED: hook fires
       → gptel-org-agent--deny-tool-calls sends denial to LLM
       → FSM continues with denial message
#+end_src

** Tag Hierarchy Pattern

Agent subtrees are identified by tags ending in =@agent=.  The tag encodes
the full delegation chain:

| Tag | Meaning |
|-----+---------|
| =main@agent= | Top-level main agent subtree |
| =researcher@main@agent= | Researcher sub-agent of main |
| =gatherer@main@agent= | Gatherer sub-agent of main |
| =researcher@researcher@main@agent= | Nested delegation (researcher of researcher) |

** Org Structure Example

With =gptel-org-use-todo-keywords= (AI/FEEDBACK keyword mode):

#+begin_src org
  ,* DOING Implement feature X                         ← user's task heading (level 1)
  ,** AI-DOING Implement feature X    :main@agent:     ← agent heading (level 2), indirect buffer A
  ,*** AI-DOING Implement feature X   :gatherer@main@agent: ← sub-agent (level 3), indirect buffer B
  ,**** AI Gatherer response                           ← sub-agent AI response (level 4)
  Gatherer results...
  ,*** AI-DOING Implement feature X   :researcher@main@agent: ← sub-agent (level 3), indirect buffer C
  ,**** AI Researcher response                         ← sub-agent AI response (level 4)
  Research findings...
  ,*** AI Main agent response                          ← AI response at agent-level + 1 (level 3)
  Based on the research results...
  ,*** FEEDBACK Follow-up question                      ← user heading at agent-level + 1 (level 3)
  What about edge cases?
  ,*** AI Edge case analysis                           ← next AI response (level 3)
  Here are the edge cases...
#+end_src

Key structural rules:
- Agent heading is one level deeper than its parent task heading
- AI response headings and FEEDBACK user headings are both at =agent-level + 1= (children of the agent heading)
- Sub-agent headings are also at =agent-level + 1=, with nested tags (e.g. =gatherer@main@agent=)
- Sub-agents do NOT get FEEDBACK headings (guarded by the =@.*@= pattern check)

Legacy tag-based structure (deprecated, for reference):

#+begin_src org
  ,* TODO Implement feature X                          ← user's task heading
  ,** :main@agent:                                     ← indirect buffer A
  ,*** :user:
  User's initial prompt...
  ,*** :assistant:
  ,**** :gatherer@main@agent:                          ← indirect buffer B
  ,***** :user:
  Prompt sent to gatherer...
  ,***** :assistant:
  Gatherer response...
  ,**** :researcher@main@agent:                        ← indirect buffer C
  ,***** :user:
  Prompt sent to researcher...
  ,***** :assistant:
  Researcher final response...
  ,*** :assistant:                                     ← main agent continues
  Based on the research results...
#+end_src

** Indirect Buffer Model

- =make-indirect-buffer= with =clone=t= shares text bidirectionally
- Emacs flattens indirect-of-indirect to root base buffer (no chaining)
- Each indirect buffer is narrowed to its agent's subtree only
- End-marker with =insertion-type=t= ensures the narrow region auto-expands
  as streaming text is inserted
- Buffer name format: =*gptel:TAG-HASH*= where HASH is a 6-character MD5
  derived from the outline path — allows multiple concurrent agents
- *Terminator heading pattern*: each task's subtree is bounded by a terminator
  child heading (FEEDBACK for AI-DOING, Results for AI-DO).  New siblings are
  inserted BEFORE the terminator to avoid disturbing existing indirect buffers.
  See [[file:gptel-indirect-buffer-ai.org]] for the full design, tracking registry,
  and Emacs indirect buffer introspection reference.
- All content inside the indirect buffer (AI responses, user prompts, sub-agent
  headings) lives at =agent-level + 1= — one level deeper than the agent heading
  itself.  This is computed by =gptel-org--compute-response-level=.
- An idempotent auto-corrector (=gptel-org--auto-correct-on-change=) is registered
  as a buffer-local =after-change-functions= hook.  It rebases any heading with
  level < =gptel-org--ref-level= and handles example block escaping and markdown
  fence conversion.  Because the correction is idempotent, sub-agent subtrees
  (already at correct levels) are automatically skipped.
- *Turn isolation*: each LLM turn (after tool execution) starts with a clean
  =:tracking-marker=.  This ensures the streaming callback treats it as a fresh
  response — the AI writes its own heading which the auto-corrector rebases.
  Without this, continuation text would land inside the previous TOOL heading's
  subtree.  See [[file:gptel-ai.org::*Turn Isolation]].

** Handover Mechanism

Handover enables escalation workflows where a triage agent gathers context
and then hands off to a specialist agent that creates a new actionable task.

#+begin_example
,**** AI-DOING Original task                          (parent TODO)
,***** :main@agent:                                   (triage agent subtree)
       Triage agent findings and tool results...
,***** :handover@main@agent:                          (handover agent subtree, folded)
       Handover agent's detailed plan...
,**** AI-DO Specialist task description               (new sibling created by handover)
       Body: handover agent's output
#+end_example

*Flow:*
1. Parent agent calls =Agent= tool with =handover=true=
2. =gptel-agent--task= extracts parent TODO content via =gptel-org-agent--extract-parent-context= and injects it into the handover agent's prompt
3. =:agent-handover t= is set on the sub-agent FSM info plist
4. Handover agent runs (reads context, uses tools, produces output)
5. On completion, =gptel-agent-subtree--handle-handover= calls =gptel-org-agent--create-handover-heading= instead of returning text
6. A new =AI-DO= sibling heading is created under the parent task
7. Parent agent receives a confirmation message (not the full output)

* Key Design Decisions

1. *Indirect buffers for isolation*: Each agent works in its own indirect
   buffer narrowed to its subtree — keeps the base buffer clean and allows
   multiple agents to run concurrently without interfering.

2. *Bidirectional text sharing*: =make-indirect-buffer= with =clone=t=
   shares text instantly between base and indirect buffers.  Changes in
   one are immediately visible in the other.

3. *No indirect chaining*: Emacs flattens indirect-of-indirect to the root
   base buffer automatically.  Sub-agent indirect buffers reference the
   same base buffer as the parent agent.

4. *Expanding end-marker*: The narrowed region's end-marker has
   =insertion-type=t=, so streaming text auto-expands the visible region
   without requiring re-narrowing after each chunk.

5. *Hash-based buffer naming*: MD5 of the outline path produces unique
   6-character hashes, preventing name collisions when multiple TODO
   headings run agents simultaneously.

6. *Read-only bypass*: =inhibit-read-only= in =create-subtree= handles
   tool confirmation overlays that set text to read-only.

7. *Context stripping by default*: Agent subtrees are stripped from the
   prompt copy during context construction.  The Advisor preset can
   override this via =gptel-org-agent-include-subtrees=.

8. *Feature flag*: Entire system gated behind =gptel-org-agent-subtrees=
   defcustom.  When =nil= (default), no behavior changes.

9. *Unconditional transform registration*: The transform function is
   registered in =gptel-prompt-transform-functions= unconditionally (via
   =with-eval-after-load 'gptel-request=); runtime guards in the function
   itself check whether to activate.  This avoids load-order race
   conditions.

10. *Tool confirmation via org headings*: In subtree mode, tool calls
    requiring user confirmation create PENDING org headings instead of the
    legacy overlay-based widget.  This integrates with org's TODO state
    change system (PENDING → ALLOWED/DENIED).  Implemented via =:around=
    advice on =gptel--display-tool-calls= — zero changes to core files.

11. *Global hash table for pending tool-calls*: Tool-call data (including
    process-tool-result closures) cannot be serialized to org properties,
    so a global hash table (=gptel-org-agent--pending-tool-calls=) maps
    unique IDs to the data.  The ID is stored as an org property
    (GPTEL_PENDING_ID) on the PENDING heading for retrieval when the user
    changes the TODO state.

12. *Overlay-free accept/deny*: Wrapper functions
    =gptel-org-agent--accept-tool-calls= and =--deny-tool-calls= take
    explicit INFO plists and do not depend on overlays or
    =gptel--fsm-last=.  They restore dynamic bindings (preset, backend,
    model) from the stored INFO.


* Org Format Instructions

Dynamic org formatting instructions are injected into the system message
at request time via prompt transforms.  This tells the LLM to respond in
org-mode format with the correct heading level for the current context.

** How It Works

1. =gptel-org-agent--transform-org-instructions= runs as a prompt transform
   (registered in =gptel-prompt-transform-functions=)
2. It resolves the response heading level dynamically:
   - In agent indirect buffers: one level deeper than the agent heading
   - In legacy subtree mode: parent heading level + 2
3. Extracts the =#+SEQ_TODO= line from the buffer if present
4. Calls =gptel-org-agent--format-instructions= to build the instruction string
5. Appends the instructions to =gptel--system-message=
6. Sets =gptel-org--org-format-response= to =t= (skips markdown→org conversion)

** Key Design Points

- The transform must run AFTER =gptel-org-agent--transform-redirect= so the
  response buffer is the final agent indirect buffer
- Instructions include a concrete example with correct heading levels,
  escaped headings (comma-prefixed), and emphasis markers for weaker models
- Gated behind =gptel-org-agent-format-instructions= variable (default =t=)

** Key Functions

| Function | Purpose |
|----------+---------|
| =gptel-org-agent--format-instructions= | Build the instruction string with examples |
| =gptel-org-agent--transform-org-instructions= | Prompt transform that injects instructions |
| =gptel-org-agent--response-heading-level= | Resolve correct heading level for context |
| =gptel-org-agent--seq-todo-line= | Extract #+SEQ_TODO from buffer |
