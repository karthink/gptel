* Overview

gptel is an asynchronous LLM request library and chat UI for Emacs. It supports
10+ backends (OpenAI, Anthropic, Gemini, Ollama, Bedrock, GitHub Copilot, Kagi,
etc.), streaming and non-streaming responses, tool/function calling, and deep
Org mode integration.

2. In the worktree, fetch and merge karthink's test changes:

   : git merge FETCH_HEAD --no-edit

3. Resolve any conflicts (typically accept =--theirs= for upstream test

   : git log --oneline FETCH_HEAD..HEAD -- <conflicting-file>

4. Push the updated subrepo branch to siroic/gptel-test *before* committing:
   : git push subrepo-test subrepo/test:main

5. Back in the main repo, commit the subrepo and clean up:
   : cd <repo-root>

*Important:* Always push the subrepo worktree branch (step 4) before running
=git subrepo commit= or =git subrepo clean=, otherwise the remote will be out
of sync with the =.gitrepo= metadata.

* Architecture

** Request Pipeline

#+begin_example
gptel-send (or gptel-request)
  → Create FSM (state machine) with state INIT
  → INIT→WAIT: create prompt buffer, apply transforms, realize query
    ├─ gptel-prompt-transform-functions (add context, apply preset)
    ├─ gptel--realize-query (parse directive, buffer→prompt list, inject media)
    └─ Convert to backend-specific JSON via cl-defmethod
  → HTTP request (Curl or url.el) — async
  → WAIT→TYPE: response received, parsed
  → TYPE→TOOL: tool calls detected → run elisp → inject results → WAIT (loop)
  → TYPE→DONE: insert response, run post-response hooks, cleanup
  → ERRS/ABRT: error handling or user abort
#+end_example

** FSM States

| State | Handler | Purpose |
|-------+---------+---------|
| INIT  | setup   | Create prompt buffer, apply transforms |
| WAIT  | request | Fire HTTP, wait for response.  Resets per-turn state (see [[*Turn Isolation]]). |
| TYPE  | route   | Decide: tool call or final response |
| TPRE  | pre-tool | Pre-tool setup, transition to TOOL |
| TOOL  | execute | Run tool functions, collect results |
| TRET  | result  | Inject tool results, display in buffer, transition to WAIT or DONE |
| DONE  | insert  | Insert response, fire hooks |
| ERRS  | error   | Handle errors |
| ABRT  | abort   | User abort cleanup |

Transition table: =gptel-send--transitions= (predicate-based).
Handlers: =gptel-send--handlers= (state→function alist).

** Key Design Patterns

- *Generic methods* (=cl-defmethod=) for backend extensibility
- *Buffer-local variables* for per-session state
- *Overlays* for response bounds and visual feedback
- *Plist-based context passing* (FSM info plist)
- *Hooks + transform functions* for customization pipeline
- *Turn isolation*: each LLM turn starts with clean insertion state.  See [[*Turn Isolation]].
- *Cache reset after inhibited edits*: code that modifies buffers inside =after-change-functions= with =inhibit-modification-hooks= t must reset the org-element cache afterward (see [[*Idempotent Auto-Corrector]])

* Preset System

Presets bundle configuration into named, reusable profiles.

- =gptel-make-preset= — Create preset with: backend, model, system prompt, tools, max-tokens, temperature, pre/post hooks
- =gptel-get-preset= — Retrieve preset by name (uses =#'equal=, works with both symbols and strings)
- =gptel--apply-preset= — Apply preset to current buffer (sets buffer-local vars)
- Presets are the foundation of the agent system (agents ARE presets)
- Keys handled by =gptel--apply-preset=: =:backend=, =:model=, =:system=, =:tools=, =:pre=, =:post=, =:parents=, plus any key matching =gptel-*= or =gptel--*= variables
- Keys explicitly skipped: =:parents= (handled via recursion), =:description=, =:pre=, =:post=, =:subagent-models= (consumed by gptel-agent layer)

* Tool System

** Defining Tools

#+begin_src elisp
(gptel-make-tool
 :name "tool_name"
 :description "What this tool does"
 :function #'my-function
 :args '((:name "arg1" :type string :description "First argument")
         (:name "arg2" :type integer :optional t :description "Optional arg"))
 :category "my-category"
 :confirm t
 :async nil)
#+end_src

** Execution Lifecycle

1. LLM returns =tool_calls= in response
2. Parser extracts tool name + arguments JSON
3. =gptel--handle-tool-use= matches tool spec
4. Confirmation if =:confirm t= (via =gptel-confirm-tool-calls=)
5. Call function with args
6. Inject tool_call + result back into messages (via =gptel--inject-prompt=)
7. Display tool results in buffer (TOOL/REASONING headings in org-mode).  In org-mode, =gptel--display-tool-results= computes the correct TOOL heading depth up front via =gptel--tool-heading-ref-level= and inserts the body (plist + =#+begin_tool … #+end_tool=) through a transient TOOL indirect buffer for proper IB isolation — see [[file:gptel-indirect-buffer-ai.org::*Integration with gptel-org.el (TOOL heading display)]]
8. TRET→WAIT transition: =gptel--handle-wait= resets per-turn state (=:tracking-marker=, =:tool-marker=, etc.) so the next LLM response starts fresh (see [[*Turn Isolation]])
9. Re-request LLM with function results
10. Loop: WAIT→TYPE→TPRE→TOOL→TRET→WAIT until no more tool calls

** Backend Tool Format

Each backend implements tool result injection differently:
- OpenAI: =tool_calls= array + =tool= role messages
- Anthropic: =tool_use= + =tool_result= content blocks
- Gemini: =functionCall= + =functionResponse= parts

*** Thinking and Reasoning Effort

gptel supports LLM thinking/reasoning features across backends:

**** Model Capabilities

Model definitions include thinking-related capabilities:
- =thinking= — model supports extended thinking (thinking blocks in response)
- =adaptive-thinking= — model supports adaptive thinking mode (model decides when/how much to think)

Models with =adaptive-thinking= capability (Claude 4.6+, Opus 4.7) have =:request-params (:thinking (:type "adaptive"))= set by default.

**** Reasoning Effort (=gptel-reasoning-effort=)

Controls how much computational effort the LLM puts into reasoning:
- =nil= — backend default
- ="low"=, ="medium"=, ="high"= — standard levels
- ="xhigh"= — extended depth (Opus 4.7 only)
- ="max"= — maximum capability

For Anthropic, this maps to the =output_config.effort= API parameter.

Can be set:
- Globally via =gptel-reasoning-effort= variable
- Per-subtree via =GPTEL_REASONING_EFFORT= org property
- Per-request via the transient menu (=-V=)
- Per-agent via agent org file =:reasoning-effort:= property

**** Thinking Auto-Injection

When =gptel-include-reasoning= is non-nil and the model has =thinking= capability:
- For =adaptive-thinking= models: =(:type "adaptive")= is used
- For older thinking models: =(:type "enabled" :budget_tokens 10240)= is used
- User can override via =:request-params= at backend or model level

The auto-injection is lowest priority — model-level =:request-params= override it via =gptel--merge-plists=.

**** Include Reasoning (=gptel-include-reasoning=)

Controls display of thinking blocks:
- =t= — include with response (default)
- =nil= — discard thinking blocks
- =ignore= — include but exclude from follow-up turns
- buffer name string — append to named buffer

Can be set per-subtree via =GPTEL_INCLUDE_REASONING= org property.
** Discussion Structure: AI/FEEDBACK Keywords

/New approach/ — replaces the classical =:assistant:= / =:user:= tag-based subtree mode.

*** Overview

Role detection uses a simple prefix rule on TODO keywords:

- *Assistant text*: heading has a TODO state with =AI-= prefix (=AI-DO=, =AI-DOING=, =AI-DONE=, etc.)
- *User text*: heading has any other TODO state (=TODO=, =DOING=, =FEEDBACK=, =DONE=, =CANCELED=, etc.)
- *Plain headings* (no TODO state): inherit role from their parent

This gives each role a distinct foreground color via =org-todo-keyword-faces= and integrates naturally with org's keyword cycling.

*** Role Detection Rule

The rule is simple and complete:

1. *Default*: everything is user text
2. *Exception*: heading with =AI-= prefixed TODO state → assistant text (including sub-headings without any TODO state)
3. *Exception to exception*: sub-heading with a non-=AI-= TODO state → user text again (including its children)

Implementation:
- =gptel-org--heading-is-assistant-p=: returns =t= when TODO state has =AI-= prefix
- =gptel-org--heading-is-user-p=: returns =t= when TODO state exists and does NOT have =AI-= prefix
- =gptel-org--restore-bounds-from-tags=: walks all headings, marks assistant subtrees with =gptel response= text property, carves out user sub-headings within, and vice versa
- Plain headings (no TODO state) return =nil= from both functions and inherit from parent via text property propagation

*** Structure Examples

**** Active task (AI processing):

#+begin_example
,* DOING Calculate 2 + 2              ← user (DOING, no AI- prefix)
,** AI-DOING Calculate 2 + 2          ← assistant (AI-DOING, has AI- prefix)
#+end_example

**** Task awaiting user feedback:

#+begin_example
,* FEEDBACK Calculate 2 + 2           ← user (FEEDBACK, no AI- prefix)
,** AI-DONE Calculate 2 + 2           ← assistant (AI-DONE, has AI- prefix)
,*** Result                           ← assistant (plain heading, inherits from AI-DONE parent)
,** FEEDBACK                          ← user (FEEDBACK, no AI- prefix)
#+end_example

**** User injecting feedback below AI heading:

,* FEEDBACK Calculate 2 + 2           ← user (FEEDBACK, no AI- prefix)
,** AI-DONE Calculate 2 + 2           ← assistant (AI-DONE, has AI- prefix)
,*** Result                           ← assistant (plain heading, inherits from AI-DONE parent)
5
,**** FEEDBACK I think result is wrong ← user (FEEDBACK, no AI- prefix)
,** FEEDBACK Calculate again           ← user (FEEDBACK, no AI- prefix)
#+end_example

**** Completed task accepted by user:

#+begin_example
,* DONE Calculate 2 + 2               ← user (DONE, no AI- prefix)
,** AI-DONE Calculate 2 + 2           ← assistant (AI-DONE, has AI- prefix)
,*** Result                           ← assistant (plain heading, inherits from AI-DONE parent)
5
,**** FEEDBACK I think result is wrong ← user (FEEDBACK, no AI- prefix)
,** DONE Calculate again               ← user (DONE, no AI- prefix)
,** AI-DONE Calculate again            ← assistant (AI-DONE, has AI- prefix)
,*** Result                           ← assistant (plain heading, inherits from AI-DONE parent)
4
,** DONE                              ← user (DONE, no AI- prefix)
#+end_example

*** Task-Level FEEDBACK Cycle

When using =gptel-org-tasks= with agents, the FEEDBACK state drives an iterative cycle at the task level:

#+begin_example
State cycle:
  AI-DO → AI-DOING → FEEDBACK → AI-DOING → FEEDBACK → ... → DONE
#+end_example

- =AI-DO= → =gptel-send= → =AI-DOING= (auto-transition on send)
- =AI-DOING= → response complete → parent heading transitions to =FEEDBACK=; a =FEEDBACK= sibling heading is created next to the agent subtree
- =FEEDBACK= → =gptel-send= → the FEEDBACK heading is mutated into a new =AI-DOING= agent subtree; parent transitions back to =AI-DOING=
- User manually sets =DONE= to end the cycle

Task-level org structure after AI response:

#+begin_example
,** FEEDBACK My task                              ← parent: awaiting user review
,*** AI-DONE Implement feature X                  ← completed agent subtree
,*** FEEDBACK                                     ← user prompt location (sibling)
#+end_example

*** Activation

Enable =gptel-org-mode= (which sets this automatically), or set
=gptel-org-use-todo-keywords= to =t= manually.  The system then:
1. Registers =AI= / =FEEDBACK= in =org-todo-keywords= with colored faces
2. Instructs the AI to start responses with =* AI <short title>=
3. Enables the idempotent auto-corrector in each agent indirect buffer (heading rebase, example escaping, fence conversion)
4. Inserts =FEEDBACK= keyword headings for user prompts after each response
5. Skips post-response title generation (the AI wrote it)

*** Idempotent Auto-Corrector

The corrector runs as a buffer-local =after-change-functions= hook (=gptel-org--auto-correct-on-change=) on each agent indirect buffer.  It fires on every buffer change and processes the changed region for corrections.

The design is stateless and idempotent:
- =gptel-org--ref-level= (buffer-local integer) is initially computed when the indirect buffer is created as =agent-level + 1=
- Headings with level < ref-level are rebased by adding =(ref-level - 1)= stars
- Headings already at or above ref-level are skipped — no double-correction possible
- Block context is checked per-heading via =gptel-org--in-example-block-p= (point-based backward search)

Corrections applied per line in the changed region:
- *Heading level rebase*: =* H= → =***** H= (only if level < ref-level)
- *Example block escaping*: =*= or =#+= at BOL inside =#+begin_example= → comma-prefixed
- *Markdown fence conversion*: =```lang= → =#+begin_src lang=, =```= → =#+end_src=

Reference level is computed by =gptel-org--compute-response-level=:
- Agent indirect buffer: first heading level + 1

Setup: =gptel-org--enable-auto-correct= is called from =gptel-org-agent--open-indirect-buffer= when creating each indirect buffer.

**** org-element cache interaction (critical pitfall)

The auto-corrector runs with =inhibit-modification-hooks= t to prevent recursive triggering from =after-change-functions=.  This has a critical side effect: the *org-element cache* does not see the secondary edits (heading rebasing, fence conversion, escaping), because =org-element--cache-after-change= is also on =after-change-functions= and is suppressed.

Without cache invalidation, =org-end-of-subtree=, =org-element-at-point=, and similar functions return stale results — they rely on =org-element-headline-parser= which computes =:end= via the cache.  For example, =org-end-of-subtree= may stop at the end of a heading line instead of spanning its children, causing =insert-user-heading= to place the FEEDBACK heading in the wrong position.

The fix: after the edit loop, when modifications were made (=modified= flag), call =org-element-cache-reset=.  A re-entrancy guard (=gptel-org--auto-correcting=) prevents the auto-corrector from being triggered by the reset.

Key rules for any code that modifies buffers inside =after-change-functions=:
- =inhibit-modification-hooks= t suppresses *all* hooks, including the org-element cache updater
- =combine-change-calls= does NOT help here — it cannot nest inside an already-running =after-change-functions= chain
- =org-element-cache-reset= is the reliable fix; it forces lazy re-parsing on next access
- Always track whether actual modifications were made to avoid unnecessary resets

*** Turn Isolation

Each LLM request/response cycle is a *turn*.  A single user interaction may
span multiple turns when tool calls are involved (WAIT→TYPE→TPRE→TOOL→TRET→WAIT→...).

*Rule*: each turn starts with clean insertion state.

=gptel--handle-wait= performs two operations on FSM info before firing each
HTTP request:

1. *Advance =:position=*: moves =:position= (start-marker) to where
   =:tracking-marker= ended up.  Without this, =:position= retains its stale
   value from the first chunk of the previous turn (right after the AI prefix
   heading), causing the next turn's content to be inserted in the middle of
   existing text — typically inside a TOOL heading's subtree.

2. *Reset per-turn keys*:

| Key | Why reset |
|-----+-----------|
| =:tracking-marker= | Forces the streaming callback to treat the response as a fresh start, inserting separator + prefix instead of continuing from the previous turn's end position |
| =:tool-marker= | Clears the consecutive-tool-result tracking so separator logic starts fresh |
| =:tool-result= | Clears stale tool results from the previous turn |
| =:tool-use= | Clears stale tool-use data |
| =:reasoning= | Clears stale reasoning state |
| =:error=, =:http-status= | Clears stale HTTP state |

*Why this matters for org-mode*: In org-mode with TODO keywords, every content
block (AI response, TOOL result, REASONING) is an org heading.  If
=:tracking-marker= persists from a previous turn, the streaming callback
(=gptel-curl--stream-insert-response=) skips prefix insertion and appends text
at the old marker position — inside the previous TOOL heading's subtree.
Resetting the markers ensures the new turn starts fresh, the AI writes its own
heading (=* AI-DOING ...=), and the auto-corrector rebases it to the correct
level.

*Marker lifecycle across turns*:
- =:position= (start-marker): created at send time, moved once per turn (when
  =:tracking-marker= is nil and prefix is inserted), then advanced by
  =gptel--handle-wait= before the next turn.  Has insertion-type nil (default).
- =:tracking-marker=: created per turn, advances with each insert (insertion-type
  t), reset to nil between turns.
- =:tool-marker=: tracks consecutive tool results within a turn, reset between
  turns.

*The uniform content model*:
- The AI writes its own headings (=* AI-DOING=, =* AI=) — gptel does not insert heading prefixes in agent indirect buffers
- TOOL results are wrapped in =* TOOL= headings by =gptel--display-tool-results=
- REASONING content is wrapped in =* REASONING= headings by =gptel--display-reasoning-stream=
- All headings are rebased to the correct level by the auto-corrector
- Each heading defines a structural boundary — no content leaks between subtrees

*** Deprecation of Classical Subtree Mode

The classical =:assistant:= / =:user:= tag-based approach (=gptel-org-infer-bounds-from-tags=) is *deprecated* and will be removed in a future release.  It remains functional as the default (=gptel-org-use-todo-keywords= defaults to =nil=) for backward compatibility.  The tag-mode fallback is still present in =gptel-org--heading-is-assistant-p= and =gptel-org--heading-is-user-p= for this reason.

Migration: set =gptel-org-use-todo-keywords= to =t= and convert existing =:assistant:= tags to =AI= keywords, =:user:= tags to =FEEDBACK= keywords.  No automated migration tool is provided yet.
