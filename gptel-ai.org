* Overview

gptel is an asynchronous LLM request library and chat UI for Emacs. It supports
10+ backends (OpenAI, Anthropic, Gemini, Ollama, Bedrock, GitHub Copilot, Kagi,
etc.), streaming and non-streaming responses, tool/function calling, and deep
Org mode integration.

The =test/= directory is a [[https://github.com/ingydotnet/git-subrepo][git-subrepo]].  When merging karthink/master,
the test subrepo may need updating separately if upstream changed its test
subproject pointer.

**** Updating tests after merging karthink/master

1. Create the subrepo branch and worktree:
   : git subrepo branch test

2. In the worktree, fetch and merge karthink's test changes:
   : cd .git/tmp/subrepo/test
   : git fetch test main
   : git merge FETCH_HEAD --no-edit

3. Resolve any conflicts (typically accept =--theirs= for upstream test
   changes, but verify local-only changes are preserved):
   : git log --oneline FETCH_HEAD..HEAD -- <conflicting-file>

4. Push the updated subrepo branch to siroic/gptel-test *before* committing:
   : git push subrepo-test subrepo/test:main
   (remote =subrepo-test= points to =git@github.com:siroic/gptel-test.git=)

5. Back in the main repo, commit the subrepo and clean up:
   : cd <repo-root>
   : git subrepo commit test
   : git subrepo clean test

*Important:* Always push the subrepo worktree branch (step 4) before running
=git subrepo commit= or =git subrepo clean=, otherwise the remote will be out
of sync with the =.gitrepo= metadata.

* Architecture

** Request Pipeline

#+begin_example
gptel-send (or gptel-request)
  → Create FSM (state machine) with state INIT
  → INIT→WAIT: create prompt buffer, apply transforms, realize query
    ├─ gptel-prompt-transform-functions (add context, apply preset)
    ├─ gptel--realize-query (parse directive, buffer→prompt list, inject media)
    └─ Convert to backend-specific JSON via cl-defmethod
  → HTTP request (Curl or url.el) — async
  → WAIT→TYPE: response received, parsed
  → TYPE→TOOL: tool calls detected → run elisp → inject results → WAIT (loop)
  → TYPE→DONE: insert response, run post-response hooks, cleanup
  → ERRS/ABRT: error handling or user abort
#+end_example

** FSM States

| State | Handler | Purpose |
|-------+---------+---------|
| INIT  | setup   | Create prompt buffer, apply transforms |
| WAIT  | request | Fire HTTP, wait for response |
| TYPE  | route   | Decide: tool call or final response |
| TOOL  | execute | Run tool functions, re-request |
| DONE  | insert  | Insert response, fire hooks |
| ERRS  | error   | Handle errors |
| ABRT  | abort   | User abort cleanup |

Transition table: =gptel-request--transitions= (predicate-based).
Handlers: =gptel-request--handlers= (state→function alist).

** Key Design Patterns

- *Generic methods* (=cl-defmethod=) for backend extensibility
- *Buffer-local variables* for per-session state
- *Overlays* for response bounds and visual feedback
- *Plist-based context passing* (FSM info plist)
- *Hooks + transform functions* for customization pipeline

* Preset System

Presets bundle configuration into named, reusable profiles.

- =gptel-make-preset= — Create preset with: backend, model, system prompt, tools, max-tokens, temperature, pre/post hooks
- =gptel-get-preset= — Retrieve preset by name (uses =#'equal=, works with both symbols and strings)
- =gptel--apply-preset= — Apply preset to current buffer (sets buffer-local vars)
- Presets are the foundation of the agent system (agents ARE presets)
- Keys handled by =gptel--apply-preset=: =:backend=, =:model=, =:system=, =:tools=, =:pre=, =:post=, =:parents=, plus any key matching =gptel-*= or =gptel--*= variables
- Keys explicitly skipped: =:parents= (handled via recursion), =:description=, =:pre=, =:post=, =:subagent-models= (consumed by gptel-agent layer)

* Tool System

** Defining Tools

#+begin_src elisp
(gptel-make-tool
 :name "tool_name"
 :description "What this tool does"
 :function #'my-function
 :args '((:name "arg1" :type string :description "First argument")
         (:name "arg2" :type integer :optional t :description "Optional arg"))
 :category "my-category"
 :confirm t
 :async nil)
#+end_src

** Execution Lifecycle

1. LLM returns =tool_calls= in response
2. Parser extracts tool name + arguments JSON
3. =gptel--handle-tool-use= matches tool spec
4. Confirmation if =:confirm t= (via =gptel-confirm-tool-calls=)
5. Call function with args
6. Inject tool_call + result back into messages (via =gptel--inject-prompt=)
7. Re-request LLM with function results
8. Loop: WAIT→TYPE→TOOL→WAIT until no more tool calls

** Backend Tool Format

Each backend implements tool result injection differently:
- OpenAI: =tool_calls= array + =tool= role messages
- Anthropic: =tool_use= + =tool_result= content blocks
- Gemini: =functionCall= + =functionResponse= parts

* Backend System

** Adding a New Backend

1. Define struct with =(:include gptel-backend)=
2. Implement =gptel--request-data= — prompt list → JSON
3. Implement =gptel-curl--parse-stream= — parse SSE/JSONL chunks
4. Implement =gptel--parse-response= — parse full JSON response
5. Implement =gptel--inject-prompt= — append message to conversation
6. Register with constructor (e.g., =gptel-make-mybackend=)

** Streaming

- Curl with =--no-buffer= for immediate chunk delivery
- SSE (Server-Sent Events) or JSONL format depending on backend
- Per-chunk callback: =gptel-curl--stream-insert-response=
- Markdown→Org stream conversion: =gptel--stream-convert-markdown->org=

*** Feasibility Analysis: TodoWrite → Org Mode TODO States
**** Current State

*Already implemented (Phase 4):* =gptel-org-agent--write-todo-org= in =gptel-org-agent.el= already converts TodoWrite calls into org headings when =gptel-org-agent-subtrees= is enabled. The mapping is:

| TodoWrite status | Org keyword |
|---|---|
| =pending= | PENDING |
| =in_progress= | RUNNING |
| =completed= | DONE |

*The gap:* The current implementation uses its own keyword set (=PENDING/RUNNING/DONE/ERROR/ABORTED=) which is completely separate from the document's =SEQ_TODO= states (=TODO/AI-DO/AI-DOING/AI-DONE/FEEDBACK/etc.=). The two systems don't talk to each other.

**** What the Referenced Tasks Propose

The vision in the two draft tasks is:

1. *User writes a =TODO= heading* → human task
2. *=gptel-send= creates =AI-DO= subheading* → queued AI task  
3. *Scheduler transitions =AI-DO → AI-DOING=* → active AI task
4. *On completion: =AI-DOING → AI-DONE=* or =FEEDBACK= if human input needed
5. *Sub-agent delegation creates nested =AI-DO= headings* under the parent
6. *Custom Org Agenda views* show human vs AI task queues separately
7. *Handoff mechanism* for triage→specialist agent escalation

***** ✅ Feasible (Medium effort)

*Auto-create =AI-DO= subheading on =gptel-send=:*

*Transition =AI-DO → AI-DOING= on task start:*

*Task queue/scheduler for AI-DO → AI-DOING:*
- Currently =gptel-send= immediately starts processing
- A scheduler would need: queue data structure, sequential/parallel execution policy, retry logic
- Would require new timer-based or idle-based dispatch mechanism
- This is a new subsystem, not just a modification

*Custom Org Agenda views:*
- Doable via =org-agenda-custom-commands= configuration
- Human view: filter on =TODO/FEEDBACK/HI-DO/HI-DONE=
- AI view: filter on =AI-DO/AI-DOING/AI-DONE=
- This is *configuration only* — suggest adding to init.el, not code changes

*Handoff mechanism (triage → specialist):*
- Requires: synthesizing context, creating new =AI-DO= heading with handoff content, killing current agent buffer
- The sub-agent delegation pattern already exists; handoff differs in that the parent terminates
- Medium-complex: needs a new tool or a modification to the =Agent= tool

***** ❌ Challenges

1. *=activeForm= has no org equivalent* — currently displayed in overlays/header-line. In org mode, this could become a property (=:ACTIVE_FORM:=) but wouldn't be as visible. Could use the heading text itself.

2. *Keyword collision* — =DONE= in agent keywords vs =AI-DONE= in document keywords. Need clear namespace separation.

** TODO Delegated sub-agents don't inherit heading-specific GPTEL_PRESET subagent-models

When a heading overrides =GPTEL_PRESET= (e.g. to =gptel-coordinator=), the main
request correctly uses that preset's model.  However, delegated sub-agents
(gatherer, executor, etc.) still resolve their models from the file-level default
preset's =subagent-models= instead of the heading-specific preset's.

The root cause is that =gptel--preset= reverts to the buffer-local (file-level)
value by the time async tool execution reaches =gptel-agent--get-parent-subagent-model=,
despite previous fixes to store/restore it via FSM info (commits =bc87c80=,
=252a46e=, =bee0621=).

*** Workaround

Define =GPTEL_PRESET= only at the top-level file property block, not per-heading.
All headings in the file will then use the same preset and its =subagent-models=
mapping.
