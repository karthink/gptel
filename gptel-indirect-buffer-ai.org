#+TITLE: gptel Indirect Buffer Management — AI Development Context
#+DESCRIPTION: Reference document for AI-assisted development of gptel-indirect-buffer.el

* Overview

The gptel indirect buffer module provides systematic management of Emacs
indirect buffers for AI task isolation in gptel's org-mode agent system.

When multiple AI tasks run concurrently (AI-DOING, AI-DO subtasks, delegated
agents), their content can merge and cursor locations get corrupted unless
each task is properly isolated in its own indirect buffer.  Sibling tasks get
merged because indirect buffers aren't properly isolating content boundaries.

The solution is the "terminator heading" pattern: each active task's subtree
is bounded by a terminator child heading (FEEDBACK for AI-DOING, Results for
AI-DO).  New sibling tasks are inserted BEFORE the terminator so existing
indirect buffers are never disturbed.

** Core Module

- File: =gptel-indirect-buffer.el=
- Provides: =gptel-indirect-buffer=
- Depends on: =org=, =org-fold=, =cl-lib=
- Declares external functions from: =gptel-org.el=, =gptel-org-agent.el=
- Complements (does NOT replace): =gptel-org-agent.el=

** Relationship to Consumers

The module is consumed by two callers in gptel, both fully migrated:

*** gptel-org-agent.el (agent subtrees)

=gptel-org-agent.el= delegates core indirect buffer operations to
this module.  The agent functions serve as thin wrappers that add
agent-specific concerns on top of the module's primitives:

- =gptel-org-agent--open-indirect-buffer= → delegates to =gptel-org-ib-create=,
  then applies agent-specific setup (flag, auto-correct, end-marker compat)
- =gptel-org-agent--close-indirect-buffer= → delegates cleanup to
  =gptel-org-ib-close=, keeps TOOL/REASONING fold logic as a pre-step
- =gptel-org-agent--setup-task-subtree= → uses =gptel-org-ib-safe-insert-sibling=
  for NEW sub-agent headings, keeps =--open-indirect-buffer= for reusing
  EXISTING subtrees

*** gptel-org.el (reasoning display)

=gptel-org.el= uses the module for the =indirect-buffer= reasoning display
mode (=gptel-org-reasoning-display= set to ='indirect-buffer=):

- =gptel-org--reasoning-create-indirect-buffer= → delegates to
  =gptel-org-ib-create= with a custom buffer name =*gptel-reasoning:BASE*=,
  then applies reasoning-specific setup (read-only, unfold subtree).
  The buffer is not displayed in any window.
- =gptel-org--reasoning-close-indirect-buffer= → delegates cleanup to
  =gptel-org-ib-close=
- The =gptel-org--reasoning-indirect-buffer= buffer-local variable is
  retained as a fast reference, but the module's registry is the source
  of truth for marker tracking

*** Module capabilities

The module provides the foundational capabilities used by both consumers:
- Terminator heading pattern (the key innovation preventing sibling merging)
- Tracking registry for all active indirect buffers
- Safe heading insertion before terminators
- Recreation after heading reorder
- Auto-correction and validation

* Architecture

** The Terminator Heading Pattern

The core innovation.  Each active task's subtree is bounded by a terminator
child heading that serves as a sentinel.  New sibling tasks are inserted
BEFORE the terminator so that existing sibling indirect buffers narrowed to
earlier positions are never disturbed.

Rules:
1. Each AI task gets its own indirect buffer narrowed to its subtree.
2. A terminator child heading bounds each task's writable content.
3. New sibling tasks are inserted BEFORE the terminator heading.
4. After reordering siblings, the moved task's indirect buffer must be recreated.

** Org Structure Example

#+begin_example
,**** AI-DOING Study feasibility ...  :main@agent:   <- Indirect-buffer 1
,***** AI-DO Subtask 1                               <- Indirect-buffer 2
,****** Results                                      <- Terminator for Subtask 1
,***** AI-DO Subtask 2                               <- Indirect-buffer 3
,****** Results                                      <- Terminator for Subtask 2
,***** FEEDBACK                                      <- Terminator for AI-DOING
#+end_example

Key observations:
- The =FEEDBACK= heading is the terminator for the AI-DOING parent task.
  It is always the LAST child.
- Each =AI-DO= subtask has its own =Results= terminator as a child.
- When a new =AI-DO Subtask 3= needs to be created, it is inserted on the
  line BEFORE the =FEEDBACK= heading — so Subtask 1 and Subtask 2's
  indirect buffers (narrowed to their subtrees) are not affected.

** Indirect Buffer Lifecycle

#+begin_example
gptel-org-ib-safe-insert-sibling (high-level convenience)
  → gptel-org-ib-ensure-terminator
    ├─ gptel-org-ib-find-terminator (search for existing)
    └─ gptel-org-ib-create-terminator (create if missing)
  → gptel-org-ib-create-heading (insert BEFORE terminator)
  → gptel-org-ib-create (create indirect buffer)
    ├─ Compute subtree region
    ├─ Create end-marker with insertion-type=t
    ├─ make-indirect-buffer with clone=t
    ├─ org-fold-core-decouple-indirect-buffer-folds
    ├─ narrow-to-region [heading-start, end-marker]
    └─ gptel-org-ib-register (add to tracking registry)

...task runs, LLM streams into indirect buffer...

gptel-org-ib-close (cleanup)
  ├─ Optional: fold subtree in base buffer
  ├─ gptel-org-ib-unregister (clean markers, remove from registry)
  └─ kill-buffer
#+end_example

** Tracking Registry

All active indirect buffers are tracked in =gptel-org-ib--registry=, a hash
table mapping buffer-name (string) to a metadata plist:

#+begin_src elisp
;; Registry entry structure:
(:buffer #<buffer *gptel:main@agent-a1b2c3*>
 :base   #<buffer project.org>
 :heading-marker #<marker at 1234 in project.org>
 :end-marker     #<marker (moves after insertion) at 1567 in project.org>
 :tag    "main@agent")
#+end_src

The registry enables:
- Looking up any IB's metadata by name
- Finding all IBs for a given base buffer
- Cleaning up dead (killed) buffer entries
- Tracking markers for recreation after reorder

** Recreation After Heading Reorder

When a subtree is reordered (e.g., by =org-move-subtree-up/down=), the
indirect buffer's narrowing restriction markers collapse because the text
at those positions was deleted and reinserted elsewhere.

The indirect buffer does NOT die — it becomes a zombie showing empty content
(point-min = point-max).  No error is raised; the condition must be actively
detected.

#+begin_example
BEFORE reorder:
  IB narrowed to [100, 200] — shows Subtask 1 content

After org-move-subtree-down:
  Text at [100, 200] was delete-and-reinserted at [250, 350]
  begv_marker and zv_marker both collapsed to position 100
  IB now shows "" (empty) — zombie state

SOLUTION: gptel-org-ib-recreate
  1. Parse buffer name → extract tag + hash
  2. Search base buffer for heading matching tag + hash
  3. Kill old IB
  4. Create new IB at heading's new position with same name
#+end_example

** Integration with gptel-org-agent.el

This section documents how =gptel-org-agent.el= delegates to the module's
primitives, and the agent-specific concerns layered on top.

*** open-indirect-buffer → gptel-org-ib-create

=gptel-org-agent--open-indirect-buffer= delegates buffer creation to
=gptel-org-ib-create=, which handles:
- Indirect buffer creation with =make-indirect-buffer=
- Narrowing to the subtree region
- Terminator heading creation (via =gptel-org-ib-ensure-terminator=)
- Registry tracking

After =gptel-org-ib-create= returns, the agent wrapper applies:
- Sets the =gptel-org-agent--flag= buffer-local variable
- Enables auto-correct mode for heading drift detection
- Sets up end-marker compatibility for streaming insertion

*** close-indirect-buffer → gptel-org-ib-close

=gptel-org-agent--close-indirect-buffer= first performs agent-specific
pre-cleanup (folding TOOL and REASONING headings), then delegates core
cleanup to =gptel-org-ib-close=, which handles:
- Removing the buffer from the tracking registry
- Killing the indirect buffer
- Cleaning up markers

*** setup-task-subtree: Two Paths

=gptel-org-agent--setup-task-subtree= uses different strategies depending
on whether the subtree is new or existing:

- *New sub-agent headings*: Uses =gptel-org-ib-safe-insert-sibling= which
  inserts a new heading /before/ the terminator heading, ensuring proper
  boundary separation.  This is the terminator-aware path.

- *Existing subtrees*: Uses =gptel-org-agent--open-indirect-buffer= (the
  wrapper around =gptel-org-ib-create=) to narrow to the already-existing
  heading.

*** On-demand Terminator Creation

Terminators are NOT eagerly created for every heading.  They are created
/on demand/ when =gptel-org-ib-create= is called (which calls
=gptel-org-ib-ensure-terminator= internally).  This means:

- A heading that is never opened as an indirect buffer has no terminator
- Terminators appear only when actually needed for IB boundary management
- =gptel-org-ib-safe-insert-sibling= also ensures a terminator exists
  before inserting the new sibling

This avoids cluttering the org tree with sentinel headings that may never
be needed, while still guaranteeing correct boundaries when they matter.

*** Agent-specific Setup (Post-creation)

After =gptel-org-ib-create= returns the indirect buffer, the agent layer
applies concerns that are outside the module's scope:

1. *Flag*: Sets =gptel-org-agent--flag= to identify the buffer as an
   agent-managed indirect buffer
2. *Auto-correct*: Enables heading drift detection and correction so that
   if a heading is moved, the IB can recover
3. *End-marker compat*: Sets up an expanding end-marker (insertion-type=t)
   at the end of the narrowed region so streaming LLM responses push the
   boundary forward rather than overflowing

** Integration with gptel-org.el (Reasoning Display)

This section documents how =gptel-org.el= uses the module for the reasoning
indirect buffer display feature (=gptel-org-reasoning-display= set to
='indirect-buffer=).

*** reasoning-create → gptel-org-ib-create

=gptel-org--reasoning-create-indirect-buffer= delegates buffer creation to
=gptel-org-ib-create= with a custom buffer name =*gptel-reasoning:BASE*=.
The module handles:
- Indirect buffer creation with =make-indirect-buffer=
- Fold state decoupling (=org-fold-core-decouple-indirect-buffer-folds=)
- Narrowing to the REASONING subtree region
- End-marker with insertion-type t for auto-expansion during streaming
- Registry tracking

After =gptel-org-ib-create= returns, the reasoning wrapper applies:
- Unfolds the REASONING heading so content is visible (=org-fold-subtree nil=)
- Sets =buffer-read-only t= (content comes from streaming, user shouldn't edit)
- Adds =kill-buffer-hook= to clear =gptel-org--reasoning-indirect-buffer= in the
  base buffer if the user kills the buffer externally
- Stores the IB reference in =gptel-org--reasoning-indirect-buffer= for quick
  access by the streaming code
- The buffer is *not displayed* in any window — like agent IBs, it exists
  only as an isolated write target

*** reasoning-close → gptel-org-ib-close

=gptel-org--reasoning-close-indirect-buffer= delegates cleanup to
=gptel-org-ib-close= which handles:
- Removing the buffer from the tracking registry
- Cleaning up markers (heading marker, end-marker)
- Killing the indirect buffer

After close, the tracking variable =gptel-org--reasoning-indirect-buffer=
is set to nil in both the current buffer and the base buffer.

*** Differences from Agent Integration

The reasoning IB is simpler than agent IBs:
- No terminator heading needed (REASONING is a single subtree, not a parent
  with sibling children)
- No auto-correct or recreation (the heading doesn't move during streaming)
- Read-only (user doesn't edit reasoning content)
- Not displayed in any window (same as agent IBs — exists only as an
  isolated write target for streaming content)
- Short-lived (exists only during streaming, closed on completion or error)

* Emacs Indirect Buffer Internals

This section documents the low-level behavior of Emacs indirect buffers,
gathered through systematic introspection.  This is the authoritative
reference for understanding the mechanics that the module relies on.

** Shared vs Independent Properties

| Property | Shared? | Notes |
|----------+---------+-------|
| Text content | SHARED | Core feature of indirect buffers |
| Text properties | SHARED | Stored in =buffer_text.intervals= |
| Modification flag / tick | SHARED | =modiff= in =buffer_text= |
| Undo list | SHARED | =buffer-undo-list= is same =eq= object |
| Markers | SHARED | All in =buffer_text.markers= chain |
| Narrowing (point-min/max) | INDEPENDENT | =begv= / =zv= per =struct buffer= |
| Point | INDEPENDENT | =pt= per buffer |
| Overlays | INDEPENDENT | Overlay interval tree per buffer |
| Major mode | INDEPENDENT | =major_mode= per buffer |
| Buffer-local vars | INDEPENDENT | =local_var_alist= per buffer |
| Mark (clone=nil) | INDEPENDENT | Separate marker objects |
| Mark (clone=t) | SHARED | Same marker object copied by ref! |
| before/after-change-functions | INDEPENDENT | But both fire on IB edit |
| Overlay mod-hooks | INDEPENDENT | Fire only in owning buffer's context |

** make-indirect-buffer Behavior

#+begin_src elisp
;; Signature:
(make-indirect-buffer BASE-BUFFER NAME &optional CLONE INHIBIT-BUFFER-HOOKS)
#+end_src

- If =BASE-BUFFER= is itself indirect, the ROOT base is used (flattening).
- =clone=t= copies major mode, minor modes, buffer-locals from base.
- =clone=nil= resets to =fundamental-mode= defaults.
- *PITFALL*: =clone=t= shares the mark-marker object by reference — setting
  mark in one buffer affects the other.

** Narrowing Behavior

- Narrowing is completely independent per buffer.
- Narrowing an indirect buffer does NOT affect the base buffer.
- Internal restriction markers: =begv_marker= (insertion-type nil) and
  =zv_marker= (insertion-type t).
- Normal =insert= at either boundary GROWS the restricted region (text absorbed).
- =insert-before-markers= at point-min PUSHES the restriction start forward
  (text excluded from the narrowed view).

** Marker Behavior Across Base/Indirect

- All markers live in the shared =buffer_text.markers= chain.
- Markers are relocated uniformly regardless of which buffer the edit occurs in.
- Marker insertion-type nil = stays before inserted text.
- Marker insertion-type t = advances past inserted text.
- Same rules apply whether insert happens in base or indirect buffer.

** What Happens When a Heading Is Moved (Reordered)

This is the scenario that motivates =gptel-org-ib-recreate=:

1. =delete-and-extract-region= moves all markers (including restriction
   markers) to the deletion point.
2. After deletion + reinsertion elsewhere, =begv_marker= and =zv_marker=
   collapse to the same position.
3. The restriction becomes EMPTY — indirect buffer shows "" with
   =point-min= = =point-max=.
4. The indirect buffer does NOT die — it becomes a zombie showing empty
   content.
5. No error is raised; the condition must be actively detected.
6. *SOLUTION*: kill and recreate the indirect buffer at the new position
   (=gptel-org-ib-recreate=).

** What Happens When Preceding Text Is Deleted

- Deletion of text BEFORE the narrowed region correctly relocates
  restriction markers.
- The narrowed content remains intact, just at shifted buffer positions.
- No special handling required.

** Indirect-of-Indirect (Chaining)

- =buffer-base-buffer= ALWAYS returns the ROOT (never an intermediate
  indirect buffer).
- Narrowing is NOT inherited through chaining — new IB starts fully widened.
- =clone=t= copies state from ROOT base buffer, not from intermediate.

** after-change-functions Firing Order

- Edit in IB: BOTH IB's and base's =before/after-change-functions= fire.
- Order: =ib-before= → =base-before= → =ib-after= → =base-after=.
- Edit in BASE: only base's hooks fire.
- =(current-buffer)= inside hooks = the buffer where the edit happened.
- *PITFALL*: base buffer hooks (auto-save, org-element cache) fire on IB
  edits.  This can trigger unexpected behavior in the base buffer.

** Overlays and Text Properties

- Overlays are completely independent per buffer.
- Base cannot see IB overlays and vice versa.
- Text properties ARE shared (stored in =buffer_text.intervals=).
- Overlay relocation IS synchronized across both buffers when text is
  inserted/deleted.
- =window= overlay property can restrict visibility to a specific window.

** Modified Flag

- Shared — cannot be set independently on base vs indirect.
- =modiff= and =save_modiff= live in =struct buffer_text=.
- =buffer-modified-tick= is also shared.
- Cannot use the modification flag to distinguish which buffer caused
  a change.

* Key Design Decisions

1. *Terminator heading pattern*: Instead of relying on complex marker
   arithmetic, a simple org heading serves as a boundary sentinel.
   Inserting before it is a single, well-defined operation that preserves
   all existing IB narrowing.

2. *Registry for tracking*: A global hash table tracks all active IBs
   with their metadata.  This enables cleanup, validation, and recreation
   without walking the buffer.

3. *Expanding end-marker (insertion-type=t)*: The end-of-subtree marker
   automatically expands the narrowed region as streaming text is inserted.
   This is the same technique used in =gptel-org-agent.el= but now
   formalized in the registry.

4. *Zombie detection over error handling*: When a heading is moved, the IB
   becomes a zombie (empty narrowing) rather than raising an error.  The
   module actively detects this condition (=point-min >= point-max=) and
   triggers recreation.

5. *Name-based recreation*: The =*gptel:TAG-HASH*= buffer name encodes
   enough information to relocate a heading after it has been moved.  The
   hash is derived from the outline path, so even after reorder the heading
   can be found by recomputing the hash.

6. *Complement, not replace*: The module does not replace the agent's
   =open/close-indirect-buffer= functions.  Instead, those functions now
   delegate core buffer management to the module (=gptel-org-ib-create=,
   =gptel-org-ib-close=) while retaining agent-specific concerns like
   flag setting, auto-correct, and TOOL/REASONING fold logic.

7. *Idempotent terminator creation*: =gptel-org-ib-ensure-terminator= finds
   an existing terminator or creates one.  Safe to call multiple times
   without duplicating terminators.

8. *Root buffer resolution*: =gptel-org-ib-base-buffer= always resolves to
   the root base buffer, matching Emacs' own flattening behavior for
   indirect-of-indirect chains.

9. *On-demand terminator creation*: Terminators are NOT eagerly created
   for every heading.  They are created on demand when =gptel-org-ib-create=
   or =gptel-org-ib-safe-insert-sibling= is called.  This avoids cluttering
   the org tree with sentinel headings that may never be needed, while
   guaranteeing correct boundaries when indirect buffers are actually used.

* Usage Examples

** Creating a Safe Sibling Task

#+begin_src elisp
;; Point is on the AI-DOING parent heading in the base buffer
(let ((ib (gptel-org-ib-safe-insert-sibling
           "AI-DO"                  ; TODO keyword
           "Research API options"   ; title
           '("researcher@agent")   ; tags
           "FEEDBACK")))           ; terminator keyword
  ;; ib is now an indirect buffer narrowed to the new AI-DO subtree
  ;; The FEEDBACK terminator was preserved as the last child
  (with-current-buffer ib
    ;; LLM can stream into this buffer safely
    (goto-char (point-max))
    (insert "Research findings...\n")))
#+end_src

** Handling a Reordered Heading

#+begin_src elisp
;; After org-move-subtree-down, check if IB is still valid
(let ((ib (get-buffer "*gptel:researcher@agent-a1b2c3*")))
  (unless (gptel-org-ib-valid-p ib)
    ;; IB narrowing has collapsed — recreate at new position
    (setq ib (gptel-org-ib-recreate ib))
    (when ib
      ;; Successfully recreated, continue using
      (message "Recreated IB at new position"))))
#+end_src

** Validating and Auto-fixing All IBs for a Buffer

#+begin_src elisp
;; Clean up dead entries first
(gptel-org-ib-cleanup-dead)

;; Validate all IBs for a specific base buffer
(dolist (name (gptel-org-ib-all-for-base (current-buffer)))
  (let* ((entry (gptel-org-ib-get name))
         (ib (plist-get entry :buffer)))
    (unless (gptel-org-ib-validate-and-fix ib)
      (message "Unrecoverable IB: %s" name)
      (gptel-org-ib-close ib))))
#+end_src

** Step-by-step: Terminator + Heading + IB Creation

#+begin_src elisp
;; Low-level approach (gptel-org-ib-safe-insert-sibling does all of this)
(save-excursion
  (org-back-to-heading t)
  ;; 1. Ensure terminator exists
  (gptel-org-ib-ensure-terminator "FEEDBACK")

  ;; 2. Create heading before the terminator
  (let ((marker (gptel-org-ib-create-heading
                 "AI-DO" "Analyze data" '("gatherer@agent") "FEEDBACK")))

    ;; 3. Create indirect buffer for the new heading
    (let ((ib (gptel-org-ib-create (current-buffer) marker)))
      ;; ib is ready for use
      ib)))
#+end_src

** Systematic Verification: Indirect Buffer as the Only Pattern

Audit performed to verify that indirect buffers are used systematically
as the /only/ design pattern for task boundary tracking in gptel-org.

*** Design Principle

Each AI action (tool launch, reasoning block, delegation, response) gets
its own isolated indirect buffer.  The indirect buffer's narrowing
(=point-min= / =point-max=) defines where the task starts and ends.
Therefore:

- *No additional marker storing* is needed for position tracking
- *No backward search* is needed to find "where am I?"
- *No ad-hoc point tracking* outside the IB narrowing

*** Verified: Legitimate Marker Usage

boundary tracking:

*** Fixed: Anti-Patterns and Redundancies

**** =gptel-org-agent--create-handover-heading= (anti-pattern → fixed)

Was: backward search from =(point-max)= to find agent heading.
Now: uses =(point-min)= of indirect buffer directly; switches to base
buffer for =org-up-heading-safe= (parent is outside narrowing).

**** =gptel-org-agent--extract-parent-context= (anti-pattern → fixed)

Was: forward scan of /entire/ base buffer to find heading by tag.
Now: uses =(point-min)= of indirect buffer as the already-known
agent heading position.  O(1) instead of O(n).

**** =gptel-org-agent--display-tool-calls= fallback (redundant → fixed)

Was: =org-back-to-heading= from =start-marker= as third-tier fallback.
Now: =(goto-char (point-min))= then =(1+ (org-current-level))= since
=point-min= IS the agent heading in an IB.

**** =gptel-org--adjust-response-headings= fallback (redundant → fixed)

Was: =re-search-forward= from =(point-min)= to find first heading.
Now: =(org-at-heading-p)= at =(point-min)= directly (in agent IB,
=point-min= IS a heading).

**** =gptel-org-agent--make-insertion-marker= (DRY helper → created)

Was: 4 copies of pos-marker creation idiom (2 with trim, 2 without — inconsistent).
Now: single helper function used by all 4 call sites with consistent behavior.


*** Defensive Fallback Removal (Registry as Single Source of Truth)

Audit performed to remove all defensive fallback patterns where code
silently fell back to =buffer-base-buffer= or =gptel-org-ib-base-buffer=
when the registry entry was missing.  The indirect buffer registry is
now the single source of truth for base buffer references.

**** Removed Patterns

| Function | Old Pattern | New Pattern |
|----------+-------------+-------------|
| =gptel-org-ib-close= | =(or (plist-get entry :base) (gptel-org-ib-base-buffer ...))= | =(plist-get entry :base)= + warning if nil |
| =gptel-org-ib-recreate= | =(or (plist-get entry :base) (gptel-org-ib-base-buffer ...))= | =(plist-get entry :base)= + early return if nil |
| =gptel-org-ib-renarrow= | =(or (plist-get entry :base) (gptel-org-ib-base-buffer ...))= | =(plist-get entry :base)= only |
| =gptel-org-ib-resolve-agent-heading= | Full base-buffer backward search fallback | Warning + nil for non-IB callers |
| =gptel-org-agent--close-indirect-buffer= | =(or (buffer-base-buffer ...) indirect-buffer)= | =(plist-get entry :base)= from registry |

**** Kept Patterns (Legitimate Usage)

These patterns are NOT defensive fallbacks and were intentionally kept:

- =gptel-org-agent--seq-todo-line=: Must access base buffer to read
  =#+SEQ_TODO:= which lives outside the narrowed region
- =gptel.el= buffer preference over =marker-buffer=: Critical defense
  against stale marker-to-buffer associations in indirect buffers
- =gptel-org-agent--ensure-tool-confirm-hook=: Intentional
  dual-registration so users can change TODO state from base buffer
- =gptel-org--reasoning-close-indirect-buffer=: Correct cleanup
  mirroring across both buffers
- Interactive commands (=gptel-org-handoff=, =find-user-task-at-point=,
  =find-indirect-buffer-at-point=): Legitimately support invocation
  from both base and indirect buffers
- =gptel-request.el= abort matching: Legitimately matches requests
  running in indirect buffers when user invokes abort from base buffer

**** Design Principle

Missing registry entries now produce explicit =gptel-org--debug=
warnings instead of being silently masked by fallback logic.  This
makes registration bugs immediately visible in debug output rather
than hidden behind defensive code that "works but is wrong."

*** Remaining Architectural Notes

- =gptel-org-agent--narrow-end-marker= (buffer-local) duplicates the
  IB registry's =:end-marker=.  Kept as a safety fallback for
  =close-indirect-buffer= in case registry entry is absent.  Could be
  removed once registry reliability is confirmed.

- =gptel-org-agent--insert-user-heading= operates in the base buffer
  (not the IB) because it inserts FEEDBACK siblings /outside/ the
  narrowed region.  This is architecturally correct — the IB pattern
  doesn't apply to operations on content outside the narrowed subtree.

- =gptel-org-agent--find-parent-indirect-buffer= forward-scans the
  base buffer for a parent tag.  This could use the IB registry
  instead but is low priority (interactive utility).
